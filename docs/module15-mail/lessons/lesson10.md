# Lesson10 Mail総復習

## ゴール

Mail総復習を、実務デバッグの中で使えるようになります。

## このLessonで身につくこと

- 何を確認するLessonなのか説明できる
- 実務で使う場面を判断できる
- 最低限の調査手順を組み立てられる
- 結果から次に確認すべきことを考えられる

## 実務ではいつ使う？

Module15 Mail / SMTP / MailHog は、アプリだけを見ても原因が分からない時に使います。

例えば以下のような場面です。

- エラーが出ているが原因箇所が分からない
- ブラウザ表示と実際のデータが一致しない
- ローカルと本番で挙動が違う
- 画面・API・ログのどこを見ればいいか迷う

## 基本

まずは「何を証拠として確認するか」を決めます。

デバッグでは、いきなり修正しません。

```text
現象
  ↓
証拠
  ↓
仮説
  ↓
検証
  ↓
修正
  ↓
再確認
```

## 実務メモ

重要なのは、コマンドを暗記することではありません。

**この確認で何が分かるのか**を説明できることです。

## よくある失敗

- 現象を具体化せずに調査する
- ログを見ずにコードだけ読む
- 1つの仮説に固執する
- 修正後の再確認をしない
- 調査メモを残さない

## ハンズオン

以下のテンプレートで、実際に1つ確認作業を行ってください。

```markdown
## 確認したこと

## 実行したコマンド・操作

## 結果

## 分かったこと

## 次に確認すること
```

## 演習

1. このLessonはどんな障害で使えますか？
2. 最初に確認する証拠は何ですか？
3. その証拠から何が判断できますか？
4. 次に確認するべきことは何ですか？

## 模範解答

現象によって変わりますが、まずはHTTPステータス、ログ、画面表示、DB、設定など、客観的な証拠を確認します。

その証拠をもとに、原因候補を減らしていきます。

## FAQ

### Q. どの順番で調査すればいいですか？

原則は、外側から内側へ確認します。

```text
Browser
  ↓
HTTP
  ↓
Web Server
  ↓
Application
  ↓
DB / Mail / External API
```
