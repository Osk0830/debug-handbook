# 22-10 条件付きブレークポイント

## なぜ条件が必要か

1000件のループで、特定IDだけ不具合が起きる場合を考えます。

```php
foreach ($users as $user) {
    syncUser($user);
}
```

通常のブレークポイントでは毎回停止します。

条件付きブレークポイントなら、対象ケースだけで止められます。

---

## Expression

ブレークポイントを右クリックし、条件を指定します。

例:

```php
$user->getId() === 123
```

配列:

```php
($item['status'] ?? null) === 'error'
```

複数条件:

```php
$user->getId() === 123 && $retryCount >= 2
```

---

## 型を意識する

```php
$id === 123
```

と

```php
$id == 123
```

は異なります。

文字列`'123'`も対象にするか、整数だけを対象にするかを決めます。

型確認:

```php
get_debug_type($id)
```

---

## Hit Count

指定回数到達したときに停止します。

例:

```text
10
```

10回目で停止。

比較条件を指定できる場合:

```text
>= 100
```

大量ループの後半だけ確認する用途に使います。

---

## Logpoint

処理を停止せず、デバッグコンソールへ情報を出します。

例:

```text
userId={userId}, status={status}
```

利用場面:

- タイミングを崩したくない
- 多数回の値を時系列で見たい
- 一時的にログを追加したい
- ソースコードを変更したくない

---

## Logpointの注意

- IDE接続中のみ機能する
- 通常ログには残らない
- 式の評価コストがある
- 大量ループで多用すると遅くなる
- 本番監視の代替ではない

---

## Exception Breakpoint

例外がthrowされた瞬間で停止します。

VS Codeの「BREAKPOINTS」領域から、例外停止設定を有効にします。

捕捉される例外も含めて停止すると、フレームワーク内部の例外が大量に発生する場合があります。

まずは未捕捉例外から始めます。

---

## 関数ブレークポイント

環境やデバッグアダプターが対応する場合、関数名を指定して停止できます。

ただしPHPでは、行ブレークポイントの方が確実で分かりやすいことが多いです。

---

## 条件式の副作用

避ける例:

```php
$repository->save($user)
```

```php
array_shift($items) !== null
```

ブレークポイント条件は何度も評価されます。

状態を変更しない式を使います。

---

## 実践例1: 特定注文

```php
$order->getId() === 98765
```

## 実践例2: 失敗時だけ

```php
$result === false
```

## 実践例3: 想定外の型

```php
!is_int($userId)
```

## 実践例4: 値が欠落

```php
!isset($payload['email']) || $payload['email'] === ''
```

## 実践例5: 特定リクエスト

```php
($_SERVER['REQUEST_URI'] ?? '') === '/api/users/123'
```

---

## 演習

次の条件で停止する式を書いてください。

1. IDが100のとき
2. ステータスが`failed`かつ再試行回数が3以上
3. 配列に`email`キーがないとき
4. 変数が整数ではないとき
5. ループの50回目だけ
