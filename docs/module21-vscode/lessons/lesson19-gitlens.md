# Lesson19 GitLensで変更理由を追う

## このLessonのゴール

- 行・ファイルの履歴から変更背景を調べられる
-  blameを責任追及ではなく調査に使える
- 標準Git機能との使い分けができる

## はじめに

障害原因は現在のコードだけでなく、「いつ、何の目的で変わったか」に隠れていることがあります。GitLensはGit履歴をエディタ内で可視化します。ただし、表示内容の正しさはGit履歴の品質に依存します。

## Current Line Blame

現在行の最終変更コミット、作者、日時、メッセージを表示できます。作者名を責任追及に使わず、関連Issueや変更目的へ到達する入口として扱います。

## File HistoryとLine History

ファイル全体の履歴と、特定行の履歴を確認できます。リファクタリングで行が移動した場合、単純なblameだけでは過去を追えないことがあるため、コミット差分も確認します。

## Commit Graph

ブランチ、マージ、タグを視覚化できます。障害がどのリリースへ含まれたか、修正がどのブランチへ入っているかを把握する助けになります。

## 標準Gitとの併用

GitLensの結果を `git log -p -- path`、`git blame`、`git show` で再確認できると、拡張が使えない環境でも調査できます。GUIの表示だけを根拠にせず、コミットIDを記録します。

## 履歴調査の限界

force push、squash、巨大コミット、意味の薄いメッセージがあると背景を追いにくくなります。履歴だけで断定せず、Issue、PR、テスト、リリース記録も確認します。

## ハンズオン

1. バグ修正対象の行でCurrent Line Blameを確認します。
2. コミットを開き、変更前後とメッセージを読みます。
3. 同じコミットをCLIの `git show` で確認します。
4. File Historyから関連変更を2件探します。
5. 「事実」「推測」「追加確認」を分けて調査メモへ残します。

## 確認問題

1. blameを責任追及に使ってはいけない理由は何ですか。
2. Line HistoryとFile Historyはどう使い分けますか。
3. 履歴だけで原因を断定できないケースを挙げてください。
