# Lesson21 GitHub Copilotをデバッグへ使う

## このLessonのゴール

- 補完・Chat・Agent系機能の役割を区別できる
- 適切なコンテキストを渡して調査を依頼できる
- 生成結果を検証し、秘密情報を守れる

## はじめに

AI支援は、コード生成よりも「調査候補を出す」「ログを読み解く」「テスト観点を増やす」用途で力を発揮します。ただし、リポジトリ固有の前提や実行結果を知らないまま断定することがあります。最終判断は証拠とテストで行います。

## 機能の違い

インライン補完は編集中の候補生成、Chatは対話的な説明・提案、Agent系機能は複数ファイルやコマンドを含む作業を支援します。利用可能な機能名やモードは更新されるため、現在のVS Codeと組織設定を確認します。

## 良い依頼の構造

現象、期待結果、再現手順、関連ファイル、エラー全文、すでに確認したこと、変更禁止範囲を伝えます。「直して」だけでは、不要な大規模変更を提案されやすくなります。

## 調査に使う

例: 「このスタックトレースから、最初に確認すべき3箇所を根拠付きで挙げて。修正コードはまだ書かない」。先に仮説を比較させると、無根拠な修正を減らせます。

## 検証手順

提案された変更は差分を1行ずつ読み、テスト、lint、型チェック、再現手順で確認します。AIが実行したと表示したコマンドも、出力と終了コードを自分で確認します。

## 情報管理

秘密鍵、顧客データ、未公開コード、個人情報を入力してよいかは組織ポリシーに従います。除外設定やEnterprise管理があっても、入力内容の分類を省略しません。

## 依存しすぎない

AIがない環境でも、検索、ログ、Debugger、Git履歴で根拠を説明できる状態を保ちます。理解できないコードを採用せず、短い変更単位でレビューします。

## ハンズオン

1. 既知のエラーについて、現象と再現手順を含む依頼文を作ります。
2. 修正ではなく、原因仮説を3つ出させます。
3. 各仮説を検索・ログ・テストで検証します。
4. 最小修正を提案させ、Git差分をレビューします。
5. AI提案なしでも説明できる調査記録を残します。

## 確認問題

1. AIへ渡すと有効なコンテキストを4つ挙げてください。
2. AIの提案を採用する前に何を確認しますか。
3. 秘密情報の扱いは何に従いますか。
