# Lesson28 チームでVS Code設定を運用する

## このLessonのゴール

- 共有設定の境界を決められる
- 推奨拡張・Task・Debugger設定をレビューできる
- 環境差をREADMEと自動化で吸収できる

## はじめに

個人のVS Code設定をそのままリポジトリへ入れると、他メンバーの環境を壊すことがあります。共有するのは「成果物や検証手順の再現性に必要な最小限」とし、個人の好みは分離します。

## 共有候補

`.vscode/settings.json`、`extensions.json`、`tasks.json`、`launch.json` が候補です。すべてを置く必要はありません。各ファイルの目的とメンテナを決めます。

## 個人設定との境界

フォーマット、lint、ファイル関連付け、共通Taskは共有向きです。テーマ、フォント、サイドバー位置、個人AI設定は共有しない方が安全です。

## レビュー観点

絶対パス、秘密情報、OS依存コマンド、未導入拡張ID、重い自動処理、不要な除外、古いDebuggerタイプを確認します。設定ファイルもコードレビュー対象です。

## READMEとの役割分担

設定ファイルだけでは前提ツールを導入できません。Node/PHP/Dockerのバージョン、初回セットアップ、Taskの用途、デバッグ開始方法、よくある失敗をREADMEへ記載します。

## CIとの整合

VS Codeのverify TaskがCIと異なるコマンドを呼ぶと、ローカルで成功してCIで失敗します。package scripts、Makefile、composer scriptsなど共通の入口を作り、双方から呼びます。

## 変更管理

拡張や設定の更新はPRで行い、影響を説明します。問題時に戻せるよう、小さなコミットにします。導入理由がなくなった設定は削除します。

## ハンズオン

1. 既存の `.vscode` を一覧し、各ファイルの目的を書きます。
2. 絶対パスと秘密情報を検索します。
3. ローカルTaskとCIコマンドを比較します。
4. 新規メンバーがREADMEだけでデバッグ開始できるか確認します。
5. 不要な共有設定を1つ削除または個人設定へ移します。

## 確認問題

1. 共有設定へ含める基準は何ですか。
2. VS Code TaskとCIで共通の入口を使う理由は何ですか。
3. 設定変更もPRレビューすべき理由を説明してください。
