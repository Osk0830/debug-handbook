# Lesson11 Debugger基礎

## このLessonのゴール

- ブレークポイントで処理を停止できる
- ステップ実行と変数確認を使い分けられる
- ログ追加よりDebuggerが適する場面を判断できる

## はじめに

Debuggerは、実行中のプログラムを一時停止し、その瞬間の変数、呼び出し履歴、条件分岐を観察する道具です。推測でログを増やす前に、停止位置と観察項目を決めると効率が上がります。

## デバッグセッションの構成

Run and DebugビューにはVariables、Watch、Call Stack、Breakpointsがあります。言語によっては専用拡張とランタイム側の設定が必要です。Node.jsは標準対応があり、PHPではXdebug、PythonではPython Debuggerなどを使用します。

## ブレークポイント

行番号の左を選択して設定します。停止しない場合は、対象コードが実行されているか、ソースマップ、パス対応、起動構成、最適化の有無を確認します。

## ステップ操作

Continueは次の停止点まで進めます。Step Overは関数内部へ入らず次の行へ、Step Intoは呼び出し先へ、Step Outは現在の関数を抜けます。すべての関数へStep Intoすると迷子になるため、疑わしい境界だけに入ります。

## VariablesとWatch

Variablesでは現在のスコープを確認します。Watchには継続して観察したい式を登録します。巨大オブジェクトを無目的に展開せず、ID、状態、条件式など仮説に必要な値へ絞ります。

## Call Stack

Call Stackは現在位置へ至る呼び出し順です。同じ関数が複数経路から呼ばれる場合、どの入口を通ったかを判定できます。フレームを選択すると、その時点のローカル変数を確認できます。

## 条件付きブレークポイントとLogpoint

特定IDだけで止める、一定回数後に止める場合は条件付きブレークポイントを使います。コードを変更せず値を記録したい場合はLogpointが便利です。ただし本番環境へ接続したデバッグは組織ルールに従ってください。

## ハンズオン

1. ループまたはAPI処理へ通常のブレークポイントを設定します。
2. Step OverとStep Intoの挙動を比較します。
3. Watchへ条件式を追加します。
4. 特定IDだけ停止する条件付きブレークポイントを作ります。
5. Call Stackから処理の入口まで戻り、経路を図にします。

## 確認問題

1. Step OverとStep Intoの違いは何ですか。
2. Watchへ登録する値はどのように選びますか。
3. 条件付きブレークポイントが有効な例を挙げてください。
