# Lesson24 ケーススタディ: PHPの500エラーを追う

## このLessonのゴール

- PHPの500エラーを層ごとに切り分けられる
- VS CodeとXdebug・ログを組み合わせられる
- 最小修正と再発防止まで整理できる

## はじめに

題材は「フォーム送信後、一部の入力でHTTP 500になる」です。500という結果だけでは、Webサーバー、PHP起動、アプリケーション、DB、外部APIのどこに原因があるか分かりません。証拠を順番に集めます。

## 再現条件を固定する

URL、HTTPメソッド、入力値、認証状態、発生時刻、環境を記録します。正常入力との差分を1項目ずつ比較します。ブラウザだけでなくcurlで再現できると、フロントエンド要因を外せます。

## ログの入口

Webサーバーのaccess/error log、PHP error log、アプリケーションログを確認します。`display_errors` を本番で有効化するのではなく、安全なログ出力先を使います。発生時刻とリクエストIDを合わせます。

## VS Codeでコードへ到達

スタックトレースのファイル名と行番号を開き、呼び出し元を参照検索します。入力値がどこで検証・変換されるかを追います。関連設定と環境変数も検索します。

## Xdebugで観察する

ローカル再現環境でブレークポイントを設定し、入力、戻り値、例外、Call Stackを確認します。Dockerでは `pathMappings` と接続先設定の不一致が典型的な原因です。

## DBと外部依存

SQLSTATE、タイムアウト、null制約、文字コード、レスポンス形式を確認します。例外を握りつぶすcatchがないか、エラーログへ十分な文脈が残るかも点検します。

## 修正と再発防止

問題入力へのガードだけでなく、入力検証、例外変換、ログ項目、テストケースを追加します。500を別の曖昧な成功レスポンスへ変えないよう、HTTPステータスと利用者向けメッセージを分離します。

## ハンズオン

1. 500エラーの再現表を作ります。
2. ログから最初のアプリケーション例外を特定します。
3. VS Codeで定義・参照・Call Stackを追います。
4. Xdebugまたは一時ログで問題値を確認します。
5. 修正、回帰テスト、ログ改善を別項目として記録します。

## 確認問題

1. 500エラー調査で最初に固定する情報は何ですか。
2. 本番でdisplay_errorsを有効にしない理由は何ですか。
3. Docker上のXdebugで確認する代表的な設定は何ですか。
