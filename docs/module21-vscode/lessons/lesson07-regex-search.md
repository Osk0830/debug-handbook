# Lesson07 正規表現検索

## ゴール

- 正規表現検索を使うべき場面を判断できる
- 文字種、繰り返し、選択、グループを使って検索できる
- 実務コードを壊さない範囲で正規表現置換を行える
- 正規表現が不向きなケースを説明できる

## 正規表現とは

正規表現は、文字列の形をパターンとして表す記法です。同じ意味でも表記が少しずつ異なる文字列をまとめて探せます。

通常検索:

```text
getUser
```

正規表現検索:

```regex
get[A-Z][A-Za-z0-9_]*
```

一致例:

```text
getUser
getProfile
getReserveData
```

VS Code Searchでは検索欄右側の`.*`を有効にします。

## まず通常検索を使う

正規表現は強力ですが、常に必要ではありません。固有のエラーメッセージ、URL、関数名が分かる場合は通常検索の方が読みやすく安全です。

正規表現が有効な例:

- 番号だけ異なるログをまとめて探す
- `console.log`と`console.error`をまとめて探す
- 複数のHTTPメソッドを探す
- 命名規則に一致する関数を探す
- 空白量が一定でないコードを探す

## 基本メタ文字

| 記号 | 意味 | 例 |
|---|---|---|
| `.` | 任意の1文字 | `a.c` |
| `\d` | 数字 | `\d+` |
| `\w` | 英数字とアンダースコア | `\w+` |
| `\s` | 空白文字 | `\s+` |
| `^` | 行頭 | `^ERROR` |
| `$` | 行末 | `;$` |
| `*` | 0回以上 | `\s*` |
| `+` | 1回以上 | `\d+` |
| `?` | 0回または1回 | `https?` |
| `{n,m}` | 回数指定 | `\d{2,4}` |
| `[]` | 文字クラス | `[A-Z]` |
| `()` | グループ | `(GET|POST)` |
| `|` | または | `error|warning` |

## 特殊文字を文字として検索する

`.`、`(`、`)`、`[`などは正規表現で特別な意味を持ちます。文字そのものを検索するにはバックスラッシュでエスケープします。

`console.log(`を探す:

```regex
console\.log\(
```

URL内のドットを探す:

```regex
example\.com
```

## 実務パターン1: デバッグ出力

JavaScript:

```regex
console\.(log|debug|info|warn|error)\(
```

PHP:

```regex
(var_dump|print_r|error_log)\s*\(
```

WordPress:

```regex
(error_log|wp_die)\s*\(
```

本番へ残してはいけないデバッグ出力の棚卸しに利用できます。

## 実務パターン2: TODOとFIXME

```regex
\b(TODO|FIXME|HACK|XXX)\b
```

Include:

```text
src/**,app/**
```

依存物や生成物を除外して、自分たちが管理するコードだけを調べます。

## 実務パターン3: HTTPメソッド

```regex
\b(GET|POST|PUT|PATCH|DELETE)\b
```

Laravelルートの例:

```regex
Route::(get|post|put|patch|delete)\s*\(
```

Expressの例:

```regex
(app|router)\.(get|post|put|patch|delete)\s*\(
```

## 実務パターン4: URL

単純な候補抽出:

```regex
https?://[^\s"'<>]+
```

このパターンは厳密なURL検証ではありません。コード中のURL候補を探す用途です。

localhostのポート:

```regex
localhost:\d{2,5}
```

## 実務パターン5: 環境変数

PHP:

```regex
env\(\s*['"][A-Z0-9_]+['"]
```

JavaScript / TypeScript:

```regex
(process\.env|import\.meta\.env)\.[A-Z0-9_]+
```

環境変数名だけでなく、どのAPIで参照されているかも確認できます。

## 実務パターン6: 数値やID

数字:

```regex
\d+
```

UUID候補:

```regex
\b[0-9a-fA-F]{8}-[0-9a-fA-F]{4}-[1-5][0-9a-fA-F]{3}-[89abAB][0-9a-fA-F]{3}-[0-9a-fA-F]{12}\b
```

メールアドレス候補:

```regex
[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}
```

これらはコード調査用の候補抽出です。入力値の完全な妥当性検証とは分けて考えます。

## 行頭・行末を使う

行頭のコメント:

```regex
^\s*//
```

行末の不要な空白:

```regex
[ \t]+$
```

空行が連続している箇所:

```regex
\n{3,}
```

複数行検索は設定や正規表現エンジンの挙動に影響されます。検索欄へ改行を含めると複数行モードになる場合があります。

## GreedyとLazy

`.*`は可能な限り長く一致しようとします。

対象:

```text
<div>one</div><div>two</div>
```

```regex
<div>.*</div>
```

は全体へ一致する可能性があります。

短く一致させる:

```regex
<div>.*?</div>
```

ただしHTMLを正規表現だけで構造的に解析するのは危険です。単純な候補抽出に限定します。

## Capture Groupと置換

検索:

```regex
user_(\d+)
```

置換:

```text
customer_$1
```

`$1`には最初のグループが入ります。

名前と値を入れ替える例:

検索:

```regex
([A-Za-z_][A-Za-z0-9_]*)\s*=\s*([^;]+);
```

置換:

```text
$2 = $1;
```

この例は構文を壊す可能性が高く、実コードへの一括適用には不向きです。Capture Groupの仕組みを理解する例として扱ってください。

## Lookaround

環境によって利用できる正規表現機能は異なります。VS Codeでは検索エンジンや設定によってLookbehindなどの対応が変わる可能性があります。

「特定文字列の前後だけを条件にしたい」場合でも、まずは通常のグループや前後コードの確認で代替できないか検討します。

## 安全な正規表現置換

1. Git作業ツリーをきれいにする、または変更を退避する
2. Searchだけ実行する
3. 一致件数と対象ファイルを見る
4. Include / Excludeを設定する
5. 代表的な一致を複数確認する
6. Replace Previewを見る
7. 少数ファイルで試す
8. テスト・Lint・型検査を実行する
9. `git diff`で意図しない変更を探す

```bash
git diff --check
git diff --stat
git diff
```

## 正規表現が不向きなケース

- 入れ子の深いHTMLやJSONを構造として変更する
- PHPやTypeScriptの構文を正確に理解してリファクタリングする
- importを安全に整理する
- 変数名をスコープ単位で変更する
- コメントや文字列を除外してコードだけ変更する

これらは言語サーバーのRename Symbol、ASTベースのツール、専用フォーマッタを優先します。

## VS Codeと`rg`で同じ正規表現を試す

```bash
rg -n 'console\.(log|warn|error)\(' src
```

```bash
rg -n '\b(TODO|FIXME)\b' . -g '!vendor/**' -g '!node_modules/**'
```

正規表現エンジンの差により、完全に同じ結果にならない場合があります。高度な記法を使うほど、対象ツールの仕様を確認します。

## ハンズオン

任意のプロジェクトで次を検索してください。

1. `TODO`または`FIXME`
2. `console.log`、`console.warn`、`console.error`
3. `http`と`https`のURL
4. 大文字スネークケースの環境変数名
5. 末尾空白

各検索について、次を記録します。

```text
検索目的:
正規表現:
Include:
Exclude:
一致件数:
誤検出:
改善後のパターン:
```

## 確認問題

1. `.`を文字として検索するにはどう書きますか。
2. `https?`は何に一致しますか。
3. `(GET|POST)`の`|`は何を意味しますか。
4. `$1`は置換時に何を表しますか。
5. HTML構造の一括変更を正規表現だけで行うべきでない理由は何ですか。

## まとめ

- 正規表現は表記揺れやパターン検索に強い
- 特殊文字は必要に応じてエスケープする
- グループと`|`で複数候補をまとめられる
- 置換ではPreview、テスト、Git差分が必須
- 構造的な変更には言語機能や専用ツールを使う
