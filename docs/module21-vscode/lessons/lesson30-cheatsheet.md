# Lesson30 VS Codeデバッグ・チートシート

## このLessonのゴール

- 調査フェーズごとの操作を素早く選べる
- ショートカットを目的と結び付けられる
- 行き詰まったときの切り分け順を再利用できる

## はじめに

このLessonはModule21の総まとめです。暗記用ではなく、障害対応中に「次に何を見るか」を決めるための索引として使います。

## 入口

`Cmd/Ctrl + Shift + P`: Command Palette

`Cmd/Ctrl + P`: ファイル検索

`Cmd/Ctrl + Shift + F`: 全体検索

`Cmd/Ctrl + Shift + M`: Problems

``Ctrl + ` ``: Terminal

## コード移動

`F12`: 定義へ移動

`Option/Alt + F12`: 定義をPeek

`Shift + F12`: 参照一覧

`Cmd/Ctrl + Shift + O`: ファイル内シンボル

`Cmd/Ctrl + T`: Workspaceシンボル

## Debugger

`F5`: 開始・継続

`F9`: ブレークポイント

`F10`: Step Over

`F11`: Step Into

`Shift + F11`: Step Out

停止しない場合は、実行経路、構成、source map、path mapping、接続先を確認します。

## 調査の基本順

1. 現象と期待結果を固定
2. 再現条件を最小化
3. Problems・ログ・Networkを確認
4. Search・Symbolsで実装へ到達
5. Debuggerで値と経路を観察
6. Git履歴で変更背景を確認
7. 最小修正
8. テストと差分確認
9. 再発防止を追加

## 行き詰まったら

対象レイヤーを変える、正常系と比較する、Empty Profileで試す、Terminalから同じコマンドを実行する、拡張を無効化する、設定スコープを確認する、キャッシュと再起動条件を確認する、という順で観測点を増やします。

## 禁止事項

根拠なく設定を大量変更しない、本番へ秘密情報や詳細エラーを表示しない、AI提案を未検証で採用しない、force操作や一括置換を差分確認なしで実行しない、調査用ログを残したままリリースしない。

## ハンズオン

1. 自分が頻繁に使う操作へ印を付けます。
2. 使っていない操作を3つ選び、実際に試します。
3. 過去の障害を1件選び、基本順へ当てはめ直します。
4. チーム固有のコマンドとログ場所を追記します。
5. 月に一度、不要になった項目を見直します。

## 確認問題

1. デバッグ調査の基本順を説明してください。
2. ブレークポイントが停止しない場合の確認項目を挙げてください。
3. 調査用ログをリリース前に削除・整理する理由は何ですか。
