# Lesson26 ケーススタディ: WordPressの表示崩れを追う

## このLessonのゴール

- WordPressのテンプレート階層とフックを調査できる
- プラグイン・テーマ・キャッシュを分離できる
- 本番を壊さずローカルで再現できる

## はじめに

題材は「特定のカスタム投稿詳細だけ、ヘッダー下のレイアウトが崩れる」です。WordPressではテンプレート、フック、ショートコード、プラグイン、キャッシュ、生成済みCSSが重なります。変更前に実際に使われている経路を特定します。

## 対象を特定する

投稿タイプ、スラッグ、使用テンプレート、ログイン状態、デバイス、キャッシュ有無を記録します。View SourceとElementsを比較し、サーバー生成時点かブラウザ適用後かを分けます。

## テンプレート階層

`single-{post_type}.php`、`single.php`、`singular.php`、`index.php` の候補を確認します。ブロックテーマではSite EditorやHTMLテンプレートも対象です。ファイル名だけで断定せず、デバッグ用コメントやQuery Monitor等で確認します。

## フックと出力元

崩れているHTMLのclass、ID、文言を検索します。`add_action`、`add_filter`、ショートコード登録、ACFフィールド参照を追います。優先度と条件分岐を確認します。

## 切り分け

子テーマ、Must-Use Plugin、通常プラグイン、キャッシュ、最適化を順に切り分けます。本番で一括無効化せず、ステージングやローカル複製を使います。

## ログとデバッグ

`WP_DEBUG_LOG` を安全な環境で利用し、画面へエラーを直接表示しない設定を検討します。PHP警告がHTMLやJSONへ混入していないか確認します。

## 修正確認

対象ページだけでなく、同投稿タイプ一覧、通常投稿、固定ページ、404、管理画面を確認します。CSSの詳細度を上げるだけの応急処置ではなく、不要なラッパーや重複出力の原因を修正します。

## ハンズオン

1. 問題URLと正常URLの条件差を表にします。
2. 実際に選択されたテンプレートを特定します。
3. 崩れているHTMLの出力元を検索します。
4. フックの登録元・優先度・条件を追います。
5. 修正後の回帰確認ページ一覧を作ります。

## 確認問題

1. WordPressでテンプレートをファイル名だけで断定できない理由は何ですか。
2. 本番でプラグインを一括停止しない理由は何ですか。
3. CSSの詳細度を上げるだけでは不十分なケースを説明してください。

## まとめ

このLessonでは、**26 ケーススタディ: WordPressの表示崩れを追う**を、単なる機能紹介ではなくデバッグの流れに組み込む方法を学びました。
