# Lesson08 Problemsパネルでエラーを集約する

## このLessonのゴール

- Problemsパネルが表示する情報の出所を説明できる
- エラー・警告・情報をファイル別に絞り込める
- Problemsに出ない障害との違いを判断できる

## はじめに

Problemsパネルは、TypeScript、ESLint、PHP Intelephenseなどの言語機能や拡張機能が検出した問題を一覧化します。ブラウザの実行時エラーやサーバーログとは異なり、主に「保存前後に静的に見つけられる問題」を扱います。

赤い波線をその場で直すだけではなく、プロジェクト全体の問題数、発生ファイル、重大度を俯瞰することで、修正漏れを減らせます。

## Problemsパネルを開く

macOSでは `Shift + Cmd + M`、Windows/Linuxでは `Ctrl + Shift + M` です。メニューの **View > Problems** からも開けます。

各項目には、重大度、メッセージ、発生元、ファイル名、行番号が表示されます。項目を選択すると該当位置へ移動します。

## エラーの発生元を読む

同じコードでも、TypeScript、ESLint、Stylelintなど複数の診断が出ることがあります。メッセージ末尾の発生元を見て、どの設定や拡張機能が報告しているか確認します。

例: `ts(2322)` はTypeScript、`eslint(no-unused-vars)` はESLintです。ルール名を検索すると、原因と修正方針へ早く到達できます。

## フィルターで調査範囲を絞る

フィルター欄ではファイル名やメッセージを絞り込めます。重大度の表示切替や、現在のファイルだけを表示する操作も有効です。大量の既存警告がある現場では、まず変更対象ファイルに限定します。

## Problemsに出ない問題

HTTP 500、SQLエラー、環境変数の不足、認証失敗、ブラウザだけで発生する例外はProblemsに出ないことがあります。Problemsが0件でも正常とは限りません。Terminal、Output、Debug Console、ブラウザDevTools、アプリケーションログへ調査を広げます。

## Quick Fixを鵜呑みにしない

電球アイコンや `Cmd + .` / `Ctrl + .` でQuick Fixを表示できます。ただし、自動修正後は必ず差分とテストを確認します。未使用変数の削除が副作用を生むなど、機械的な修正が意図と一致しない場合があります。

## ハンズオン

1. 意図的に未使用変数、型不一致、未定義関数を含むファイルを用意します。
2. Problemsを開き、重大度と発生元を確認します。
3. 現在のファイルだけに絞り込みます。
4. Quick Fixを適用する前後でGit差分を確認します。
5. Terminalでlintや型チェックを実行し、Problemsの件数と比較します。

## 確認問題

1. Problemsと実行時ログの違いは何ですか。
2. 同じ行に複数の問題が出たとき、最初に何を確認しますか。
3. Quick Fix後に必要な確認を2つ挙げてください。
